title: 'Connect Your Database'
---

**Your database is the system of record — Ablo never hosts your data.** Every
schema model is backed by your own Postgres; Ablo is the transaction layer on
top of it. There are two ways to connect, and they are the same product with the
same writes — the only difference is where your database credential lives:

| | How Ablo reaches your Postgres | Use when |
| --- | --- | --- |
| **Connection string** (default) | You pass `databaseUrl` to `Ablo(...)`; Ablo registers the connection and commits each write directly, behind row-level security. | You can hand over a scoped connection string. |
| **Signed endpoint** | Your app exposes one route built from an ORM adapter; Ablo sends signed commit requests and your app writes its own database. | Database credentials must never leave your infrastructure. |

Customer apps must define an Ablo schema. The schema is the contract between
the SDK, agents, realtime subscriptions, and your database — for the **synced**
models only. Use `defineSchema`, `model`, and Zod alongside your existing ORM
schema; the Ablo schema describes the collaborative tables, your own
Drizzle/Prisma schema keeps describing everything else (auth, billing, anything
not synced). One database, two schemas, side by side.

What Ablo stores, in both shapes: your schema *definition* (model names,
fields, types — pushed with `ablo push`), your hashed API keys, a safe
projection of the connection registration (host, database, schema — the
connection string itself is sealed and never returned by any API), and the
commit log that drives sync. Never your rows.

Both shapes use the same SDK call:

```ts
await ablo.tasks.create({ data: { title: 'Draft launch plan', status: 'todo' } });
await ablo.tasks.update({ id: 'task_123', data: { status: 'done' } });
const task = ablo.tasks.get('task_123');
```

Multiplayer behavior is the same in both shapes. Writes made through
`ablo.<model>.create/update/delete` are coordinated by Ablo, then confirmed rows
fan out to subscribers. Database writes that bypass Ablo need outbox events so
connected humans and agents see the change.

## Connection String (default)

Pass `databaseUrl` straight to the client. On `ready()` — before the first sync
— the SDK registers that connection as your project's store, and Ablo commits
every confirmed write directly to your database:

```ts
const ablo = Ablo({
  schema,
  apiKey: process.env.ABLO_API_KEY,
  databaseUrl: process.env.DATABASE_URL, // your Postgres — rows live here, never with Ablo
});

await ablo.ready();
```

`databaseUrl` defaults to `process.env.DATABASE_URL`, so a server-side app that
already exports it needs nothing extra. The connection string is sent once over
TLS, stored sealed, and never returned by any API.

Provision your synced-model tables in that database:

```bash
npx ablo migrate            # provision your synced models in DATABASE_URL
npx ablo migrate --dry-run  # preview the exact SQL first
```

`ablo migrate` only provisions the models in your Ablo schema (with row-level
security already applied). It never touches your other tables — auth, billing,
anything outside the Ablo schema stays owned by your own ORM migrations. See
the [CLI](/cli) for the full migrate flow and Zod-to-Postgres type mapping.

<Warning>
  `databaseUrl` is **server-only** — it carries your database credentials. Constructing a client
  with it in a browser throws, and (unlike `apiKey`) `dangerouslyAllowBrowser` does **not** override
  that. Register your database from a server runtime.
</Warning>

<Note>
  Give Ablo a **non-superuser, non-`BYPASSRLS` role**, not your owner connection. Ablo runs your
  tenant plane with row-level security **forced**, and rejects a role that could bypass it. A
  restricted role is what lets the database — not just app code — guarantee isolation. Connection
  strings resolving to loopback or private address ranges are rejected too.
</Note>

## Signed Endpoint

When a connection string must not leave your infrastructure, keep
`DATABASE_URL` in your app and expose one HTTPS endpoint instead: Ablo sends
signed requests to your server, your server writes its database, and Ablo
coordinates the confirmed stream. Omit `databaseUrl` from `Ablo(...)` in this
shape — the endpoint *is* the connection, and registering both would point Ablo
at your database twice.

You expose one endpoint that verifies Ablo's signed request before opening a
database transaction. The contract is implemented and tested:

- `dataSourceNext(...)` verifies signed requests before your adapter runs.
- The adapter (`prismaDataSource` / `drizzleDataSource`) handles reads, commits,
  idempotency, and the outbox events feed.
- Endpoint commits append confirmed rows into Ablo's commit log for realtime
  fan-out.
- Outbox events are polled and deduped by event id.
- SDK-origin outbox echoes are filtered by `clientTxId`.

Choose this shape when your existing service layer already owns writes, the
data is regulated, or the database must stay behind your own API. If you are
migrating an app where every button already calls a backend endpoint, read
[Integration Guide](/integration-guide) first, then
[Existing Python Backend](/examples/existing-python-backend) for a concrete
service-owned database example.

### What the integration has

| Field                | Purpose                                                                              |
| -------------------- | ------------------------------------------------------------------------------------ |
| Data Source endpoint | The public HTTPS endpoint in your app that Ablo calls.                               |
| API key              | Stored in your app as `ABLO_API_KEY`; used by the SDK and the Data Source endpoint.  |
| External-write feed  | The adapter's `ablo_outbox` table, served as the events feed Ablo polls.             |
| Status               | Last successful request, last error, and delivery attempts.                          |

The shape is the same as a production webhook integration:

1. Expose one Data Source endpoint in your app.
2. Store `ABLO_API_KEY` in your app.
3. Verify signed HTTP calls before opening a database transaction.
4. Keep your database credentials in your app.
5. Write an outbox row when data changes outside Ablo.

### Route

Wire your ORM through an adapter. The adapter knows how to read rows, run the
commit transaction, dedupe by `clientTxId` (idempotency), and serve the outbox
events feed — so the route is a one-liner:

```ts
// app/api/ablo/source/route.ts
import { dataSourceNext } from '@abloatai/ablo/source/next';
import { prismaDataSource } from '@abloatai/ablo/source';
import { schema } from '@/ablo.schema';
import { prisma } from '@/db';

export const runtime = 'nodejs';

export const { POST } = dataSourceNext({
  schema,
  apiKey: process.env.ABLO_API_KEY!,
  adapter: prismaDataSource(prisma, schema),
});
```

On Drizzle, swap one line — the adapter takes the Drizzle db and your Ablo
`schema` (not your table objects):

```ts
import { drizzleDataSource } from '@abloatai/ablo/source/drizzle';
// ...
  adapter: drizzleDataSource(db, schema),
```

The adapter only touches the synced models named in your Ablo `schema`. The rest
of your database — auth, billing, any non-synced tables — stays owned by your
own ORM schema and migrations.

Your app code still writes through the normal model API:

```ts
await ablo.tasks.update({
  id: 'task_123',
  data: { status: 'done' },
  wait: 'confirmed',
  readAt: snap.stamp,
  onStale: 'reject',
});
```

### Commit Request

When Ablo calls your Data Source, it sends a signed JSON request:

```ts
{
  type: 'commit',
  clientTxId: 'tx_...',
  operations: [
    {
      type: 'UPDATE',
      model: 'tasks',
      id: 'task_123',
      input: { status: 'done' },
      readAt: 1042,
      onStale: 'reject',
    },
  ],
  scope: {
    participantId: 'agent:triage',
    participantKind: 'agent',
    organizationId: 'org_123',
    requiredSyncGroups: ['org:org_123'],
    mode: 'live',
  },
}
```

Return canonical rows:

```ts
{
  rows: [
    { id: 'task_123', title: 'Fix docs', status: 'done' },
  ],
}
```

Use explicit `deltas` only when your source already computes canonical change
events.

### External Writes

If your app changes data outside Ablo — old endpoints, cron jobs, admin tools,
imports — write a row into the adapter's `ablo_outbox` table in the same
transaction as the change. The adapter serves those rows as the events feed Ablo
polls, so connected humans and agents stay current:

```ts
// inside your own (non-Ablo) write transaction
await tx.abloOutbox.create({
  data: {
    model: 'tasks',
    entityId: task.id,
    type: 'UPDATE',
    data: task,
    organizationId: task.organizationId,
    // include the originating clientTxId if this change came from an Ablo commit
  },
});
```

`clientTxId` lets Ablo drop SDK echoes that already produced a realtime update.
Outbox rows without `clientTxId` are treated as external writes. The
`ablo_outbox` and `ablo_idempotency` tables are provisioned by `ablo migrate`
(or the adapter's own setup) — they live alongside your application tables.

### Production Checklist

Before using the signed-endpoint shape in production:

- Keep `DATABASE_URL` in the customer app or backend environment.
- Use only the Data Source endpoint and `ABLO_API_KEY` as the customer-facing integration boundary.
- Verify signatures before opening a database transaction.
- Store `clientTxId` in an idempotency table before applying writes.
- Return canonical rows after each commit.
- Write outbox events in the same transaction as non-Ablo writes.
- Dedupe outbox events by event `id`.
- Monitor last success, last error, retry count, event lag, and cursor.

## Security

- Verify endpoint requests with `ABLO_API_KEY`.
- Keep database credentials in server runtimes (your app, or the sealed Ablo registration).
- Dedupe commits by `clientTxId`.
- Dedupe external events by event `id`.
- Use HTTPS in production.

The API key is not a database credential. It only lets your route verify that
the request came from Ablo and was not modified in transit.
