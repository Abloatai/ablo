Use this guide when you are adding Ablo to a real product, not a demo.

The normal integration is one client:

```ts
import Ablo from '@abloatai/ablo';
import { defineSchema, model, z } from '@abloatai/ablo/schema';
```

Declare the models Ablo coordinates, then read and write through
`ablo.<model>`. React, server actions, backend workers, and agents should all use
that same model path.

```txt
schema -> ablo.<model>.retrieve(...) -> ablo.<model>.update(...)
```

Capabilities, tasks, commits, and receipts exist under the hood. Most apps do
not create them by hand.

## Pick The Backing Mode

Every schema model has a backing store. The SDK call shape stays the same.

| Mode | Rows live in | Use when |
|---|---|---|
| Data Source | Your app database | Your existing service writes the database and stays source of truth. This is the default existing-backend path. |
| Direct Postgres connector | Your Postgres | Optional convenience connector. You explicitly let Ablo open a database connection and provision tables. |
| Schema-less resource API | Custom runtime | A server worker, MCP route, or migration script intentionally cannot import the app schema. |
| Ablo-managed | Ablo | [Roadmap](/roadmap). Zero-database hosted storage, not yet available. |

For an existing backend, expose a signed Data Source endpoint and keep
`DATABASE_URL` inside your app. Application and agent code authenticate with
`ABLO_API_KEY`; Ablo calls your endpoint, your server writes the database, and
Ablo coordinates the resulting stream.

Use the direct Postgres connector only when you want Ablo to own the database
connection and provisioning flow.

## Test With Sandboxes

Use the public `/sandbox` page to understand the state flow. It is a visual,
deterministic demo; it does not call your API key or mutate hosted Ablo data.
It is also built for coding agents: copy the sandbox prompt into Claude Code or
Codex and ask it to wire one real resource through the schema model API.

Use the authenticated org dashboard sandbox for real integration work. The
default sandbox is the equivalent of Stripe test mode:

- it is scoped to the organization,
- it has an isolated sync group prefix,
- it mints `sk_test_*` keys,
- it can be reset without touching live state,
- additional sandboxes can start blank or from copied live configuration.

Live keys and sandbox keys are separate. Use `sk_test_*` while wiring your app,
agents, and Data Source endpoint; move to `sk_live_*` only when the same schema
and write path are ready for production.

When handing this to a coding agent, give it a concrete target:

```txt
Add Ablo Sync to this app for one resource that humans and agents both edit.
Use the org sandbox sk_test_* key. Declare schema, add the Ablo client, replace
one write with ablo.<model>.update({ id, data, wait: 'confirmed' }), and add a
smoke test for two concurrent writers.
```

## 1. Declare A Schema

Start with fields and relations. Keep load strategies, indexing hints, and
read-only/mutable shortcuts out of the first version unless you already need
offline-heavy local cache behavior.

```ts
// src/ablo.schema.ts
import { defineSchema, model, z } from '@abloatai/ablo/schema';

export const schema = defineSchema({
  tasks: model({
    id: z.string(),
    projectId: z.string(),
    title: z.string(),
    status: z.enum(['todo', 'doing', 'done']),
    assigneeId: z.string().nullable(),
    updatedAt: z.string(),
  }),
});
```

## 2. Create The Client

Trusted runtimes can use `ABLO_API_KEY`.

```ts
// src/ablo.ts
import Ablo from '@abloatai/ablo';
import { schema } from './ablo.schema';

export const ablo = Ablo({
  schema,
  apiKey: process.env.ABLO_API_KEY,
});
```

Browser apps must **never** ship the server `apiKey`. The browser gets its own
short-lived credential: build an `Ablo` client whose `getToken` fetches a token
from a backend route, and pass that **client instance** to `<AbloProvider>`.

First, a backend route mints a per-user session token (an `ek_`) using the
server client. Authenticate it with your own session before minting.

```ts
// app/api/ablo-session/route.ts
import { ablo } from '@/ablo'; // server client, holds ABLO_API_KEY
import { getCurrentUser } from '@/auth';

export async function POST() {
  const user = await getCurrentUser(); // your existing auth
  if (!user) return new Response('Unauthorized', { status: 401 });

  const session = await ablo.sessions.create({ user: { id: user.id } });
  return Response.json({ token: session.token });
}
```

Then point a browser `Ablo` client at that route and hand the instance to the
provider. `getToken` is called for the first mint and every refresh — the
secret `apiKey` never enters the bundle.

```tsx
// app/providers.tsx
'use client';

import Ablo from '@abloatai/ablo';
import { AbloProvider } from '@abloatai/ablo/react';
import { schema } from '@/ablo.schema';

// Build once at module scope — a new instance per render tears down the socket.
const ablo = Ablo({
  schema,
  getToken: () =>
    fetch('/api/ablo-session', { method: 'POST' })
      .then((r) => r.json())
      .then((d) => d.token),
});

export function Providers({ children }: { children: React.ReactNode }) {
  return <AbloProvider client={ablo}>{children}</AbloProvider>;
}
```

For a read-only browser experience with no session route, use a publishable
`pk_` key directly instead: `Ablo({ schema, apiKey: 'pk_...' })`.

## 3. Read State

Use `retrieve` (async) when the row may not already be local — it goes to the
server and resolves with the row.

```ts
await ablo.ready();

const task = await ablo.tasks.retrieve({ id: 'task_123' });
if (!task) throw new Error('task not found');
```

Use `get`, `getAll`, and `getCount` for synchronous local reads after data has
loaded.

```ts
const task = ablo.tasks.get('task_123');
const activeTasks = ablo.tasks.getAll({
  where: { projectId: 'proj_123' },
  orderBy: { updatedAt: 'desc' },
  limit: 50,
});
```

In React, selector `useAblo` is the public read API. Selectors run on every
local change, so they must use the **synchronous** reads (`get`, `getAll`,
`getCount`) — never `retrieve`/`list`, which are async:

```tsx
'use client';

import { useAblo } from '@abloatai/ablo/react';

export function TaskRow({ task: serverTask }: { task: Task }) {
  const task = useAblo((ablo) => ablo.tasks.get(serverTask.id)) ?? serverTask;

  return (
    <button disabled={task.status === 'done'}>
      {task.title}
    </button>
  );
}
```

Use zero-argument `useAblo()` only in callbacks and effects:

```tsx
const ablo = useAblo();
```

## 4. Write State

For simple writes:

```ts
await ablo.tasks.update({
  id: 'task_123',
  data: { status: 'done' },
  wait: 'confirmed',
});
```

For writes based on state the user or agent already read, snapshot first and
reject stale updates:

```ts
const snap = ablo.snapshot({ tasks: 'task_123' });

await ablo.tasks.update({
  id: 'task_123',
  data: { status: 'done' },
  readAt: snap.stamp,
  onStale: 'reject',
  wait: 'confirmed',
});
```

`wait: 'confirmed'` resolves after the server accepts the write. Rejections roll
back optimistic local state and throw a typed `AbloError`.

## 5. Multiplayer Is Automatic

There is no separate multiplayer setup.

If humans, server actions, and agents use the same schema model resource, they
share the same stream:

```txt
human UI -> ablo.tasks.update(...)
agent    -> ablo.tasks.update(...)
server   -> ablo.tasks.update(...)
```

Ablo coordinates those writes, fans out confirmed deltas, coordinates exclusive
access through claims, and lets callers reject stale writes with `readAt`.

Direct writes to your own database bypass that stream until your app reports the
change through Data Source events.

## 6. Existing API Backend

This is the path for a product where buttons already call Python, Rails, Go, or
Node endpoints.

Keep your backend and database canonical. Add Ablo as the shared write path for
the records that need multiplayer now and agent-safe writes later.

```txt
Button
  -> ablo.tasks.update(...)
  -> Ablo
  -> signed Data Source request
  -> existing backend service
  -> app database
  -> Ablo realtime fanout
```

The migration can be gradual:

1. Declare schema for one resource, such as `tasks`.
2. Keep existing server loads for first paint.
3. Add `useAblo((ablo) => ablo.tasks.get(id)) ?? serverTask` for live rows.
4. Add one Data Source endpoint that calls the existing service layer.
5. Move one mutation button from `fetch('/api/tasks/...')` to `ablo.tasks.update(...)`.
6. Add an outbox/events path for writes that still happen outside Ablo.
7. Let agents use the same `ablo.tasks.retrieve(...)` and `ablo.tasks.update(...)`.

For the full Python shape, see
[Existing Python Backend](/examples/existing-python-backend).

## 7. Data Source Endpoint

Use a Data Source when your app database remains the source of truth. Wire your
ORM through an adapter; it handles reads, the commit transaction, idempotency,
and the outbox/events feed:

```ts
// app/api/ablo/source/route.ts
import { dataSourceNext } from '@abloatai/ablo/source/next';
import { drizzleDataSource } from '@abloatai/ablo/source/drizzle';
import { schema } from '@/ablo.schema';
import { db } from '@/db';

export const runtime = 'nodejs';

export const { POST } = dataSourceNext({
  schema,
  apiKey: process.env.ABLO_API_KEY!,
  adapter: drizzleDataSource(db, schema),
});
```

On Prisma, swap the adapter for `prismaDataSource(prisma, schema)` from
`@abloatai/ablo/source`. The adapter is given your Ablo `schema`, so it only
touches the synced models; the rest of your database stays yours.

Ablo needs your Data Source endpoint and API key. Your app stores one Ablo
credential:

```bash
ABLO_API_KEY=sk_live_...
```

The API key verifies Ablo's request. It is not a database credential.

## 8. Agents

Agents should use the same model methods as the app when they can import the
schema.

```ts
// Take a claim so the agent has exclusive, ordered access while it works.
await using claim = await ablo.tasks.claim({ id: taskId, action: 'update' });
if (!claim.data) return;

await ablo.tasks.update({
  id: taskId,
  data: { status: 'done', summary: await summarize(claim.data) },
  wait: 'confirmed',
});
// claim auto-releases at end of scope
```

Use AI SDK for the model loop. Put Ablo inside the tool that persists the final
change.

```ts
import { tool } from 'ai';
import { z } from 'zod';

const completeTask = tool({
  description: 'Mark a task done with a summary',
  inputSchema: z.object({
    taskId: z.string(),
    summary: z.string(),
  }),
  execute: async ({ taskId, summary }) => {
    return ablo.tasks.update({
      id: taskId,
      data: { status: 'done', summary },
      wait: 'confirmed',
    });
  },
});
```

Use the schema-less protocol client (`ablo.model(name)` + `commits.create(...)`)
only for custom server runtimes that intentionally cannot import the schema.

## Optional Surface

| Optional piece | Why it exists |
|---|---|
| `/react` | Live React selectors, provider lifecycle, presence, sync status. |
| `/testing` | Test harnesses and deterministic mocks. |
| `Data Source` | Keep your app database canonical. |
| `persistence: 'indexeddb'` | Durable browser cache and offline queueing for apps that need it. |
| `claim` | Take exclusive, ordered access to a row; queue concurrent writers. |
| `snapshot` + `readAt` | Reject writes based on stale state. |
| `mutable`, `readOnly`, `field`, `indexed` | Advanced schema and local-cache tuning. |
| `resource(...)` and `commits.create(...)` | Low-level protocol access for custom runtimes. |

The first integration should not need most of these. Start with schema and
model methods, then add the optional pieces where the product actually needs
them.

## Method Cheatsheet

| Method | Use it for |
|---|---|
| `retrieve({ id })` | Async fetch of one row from the server. |
| `list({ where, orderBy, limit })` | Async server query. |
| `get(id)` | Synchronous local read of one loaded row. |
| `getAll(options?)` | Synchronous local collection read. |
| `getCount(options?)` | Synchronous local count. |
| `create({ data, id? })` | Create through the model resource. |
| `update({ id, data, ...options })` | Update through the model resource. |
| `delete({ id, ...options })` | Delete through the model resource. |
| `claim({ id, action?, ttl? })` | Take an `await using` claim for exclusive, ordered access. |
| `claim.state({ id })` | See the current holder and queue. |

Do not use `ablo.commit` as the first write API. Most callers should never need
the low-level commit plane directly.
